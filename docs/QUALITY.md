# 质量与测试策略

目标不是为了测试而测试，而是为了收敛：避免一路朝错误方向狂奔。

本项目适合采用轻量的测试驱动或验收驱动。严格 TDD 适合稳定代码；但比赛项目还包含调研、产品演示、Notebook 和快速迭代。因此本项目的核心规则是：

> 先定义证据，再开始构建。

## 调研与比赛库

调研类工作用“来源和字段完整性”作为质量门槛。

必要证据：

- 官方来源链接；
- 参赛资格标签；
- 截止时间或时间窗口；
- 8GB 可行性；
- 推荐级别和理由；
- 比赛库校验通过。

命令：

```powershell
pwsh scripts/check-env.ps1
pwsh scripts/export-competition-index.ps1
pwsh scripts/check-registry.ps1
```

## 应用赛

应用赛采用验收驱动开发。

实现前先写清：

- 目标用户；
- 一个核心任务；
- 成功标准；
- 演示脚本；
- 失败场景；
- 提交物清单。

最低测试/评测：

- 冒烟测试：应用能按文档命令启动；
- 黄金任务：一个已知场景能产出可接受结果；
- RAG / Agent 回答有来源或可解释依据；
- UI 有截图或短演示视频；
- 提交清单已完成。

## 算法赛

算法赛采用实验驱动开发。

建模前先写清：

- 评价指标；
- 验证集划分方式；
- 数据泄露风险；
- baseline 计划；
- 第一次提交目标。

最低测试/评测：

- 数据加载冒烟测试；
- 验证集划分可复现；
- baseline 分数已记录；
- 提交文件格式检查；
- 实验日志已更新。

## 什么时候用严格 TDD

适合严格 TDD 的内容：

- 可复用脚本；
- 数据校验器；
- 解析器；
- 评分/评测工具；
- 提交应用中的 API 接口。

适合轻量验收检查的内容：

- 探索性 Notebook；
- 短周期 Hackathon 原型；
- 一次性 demo；
- UI 细节打磨。

## 停止并重评的信号

出现以下情况时停止推进，先重评方向：

- 参赛资格无法确认；
- 前 48 小时交付物说不清；
- 多轮实验后验证分数没有改善；
- 应用无法稳定演示一个端到端场景；
- 所需算力超出 `A/B/C` 可行范围。
